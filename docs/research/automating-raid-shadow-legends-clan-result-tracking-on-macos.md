## How each extraction path looks on macOS

### Screen capture plus on-device OCR

This is the cleanest option on your platform. AppleŌĆÖs **ScreenCaptureKit** is designed for high-performance capture of displays, apps, and windows on macOS, and Apple exposes window/app selection primitives through `SCShareableContent` plus single-frame capture through `SCScreenshotManager`. AppleŌĆÖs **Vision** framework provides `VNRecognizeTextRequest` and a dedicated ŌĆ£Recognizing Text in ImagesŌĆØ flow for OCR. ### Log-file inspection

This is the safest exploratory path before you write any code against the live client. PlariumŌĆÖs own support documentation says that on Mac, Plarium Play stores logs under `\Users\username\Public\PlariumPlay\logs`. Those logs are officially documented for troubleshooting rather than structured gameplay export, so I would not assume they contain Hydra/Chimera/Siege result tables. ### Network proxying

This is plausible but brittle. Tools like **Proxyman** and **Charles** can capture and decrypt HTTP/HTTPS traffic on macOS when the local machine trusts the debugging CA certificate; Charles explicitly describes SSL proxying as man-in-the-middle HTTPS inspection, and Proxyman documents HTTPS request/response interception on macOS. The problem is that apps using **certificate pinning** may reject the proxy certificate and fail to connect. Charles documents that behavior directly, and ProxymanŌĆÖs troubleshooting docs say some protected apps cannot be intercepted for exactly this reason. ## The architecture I would actually build

I would build a **small native macOS companion app** with a deliberately narrow job: detect that the RAID window is visible, capture only the relevant window or region, OCR the known result blocks, normalize the text into your existing JSON schema, and push it to your website and Telegram formatter. AppleŌĆÖs screen-capture and OCR APIs are already enough for that pipeline, and this keeps the system **read-only** from the gameŌĆÖs point of view. I would split the workflow into two modes. ## Risk boundaries and what I would not do

I would **not** build against binary patching, injected libraries, or a Unity-hooking workflow on your main account. That is the zone most likely to drift into the reverse-engineering conduct Plarium explicitly forbids, and it creates ongoing maintenance work every time the client updates. The same goes for taking the HellHades remote-extraction model and repurposing it for clan data: even if the service says it discards credentials afterward, handing account credentials and live 2FA into a remote queue is a very different trust model from a local, read-only capture helper.
